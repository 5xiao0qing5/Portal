# 接手告诫

这份文档是写给下一个接手 `Portal` 项目的人。

先说结论：

**不要一上来就“大改”或者“重写全部”。**
这个项目不是单纯的 Android App，而是 **前台 App + Xposed/LSPosed Hook + 系统行为伪造** 的组合体。很多代码看起来脏，但里面有不少是为了兼容不同 ROM、不同定位链路、不同 Hook 时机硬堆出来的“经验代码”。贸然整理，极容易把“能用”改成“看起来很整洁但已经失效”。

---

## 1. 先理解：哪些地方是“屎山核心”，不能乱碰

### 1.1 `Portal/app`
前台控制层，负责：
- 地图展示
- 搜索地点
- 选点/路线编辑
- 设置页
- 和 Xposed 模块通信
- 悬浮窗/摇杆/路线模拟控制

这一层问题最多：
- 历史硬编码多
- 字符串资源曾经有过编码污染
- `Fragment + ViewBinding + SharedPreferences` 之间耦合很重
- 很多状态是“页面逻辑 + ViewModel 状态 + 本地配置”混在一起

### 1.2 `Portal/xposed`
系统 Hook 层，负责：
- 位置 Hook
- GNSS/NMEA Hook
- 传感器 Hook
- 基站/Wi‑Fi Hook
- 和前台 App 的 `sendExtraCommand()` 协议

这层最危险：
- 一旦改坏，前台 App 看起来可能完全正常，但系统侧已经不工作
- 出问题时通常不是编译错误，而是“功能失效”或“某些 App 失效”
- 不同 ROM 行为差异巨大，很多兼容逻辑不能凭直觉删

### 1.3 `MockServiceHelper` 和 `FakeLoc`
这是“中枢神经”：
- `MockServiceHelper`：App -> Xposed 命令桥
- `FakeLoc`：Xposed 侧状态中心

**如果你要加配置项、改默认行为、加开关，优先看这两个。**
但不要随意改命令名、字段名、布尔语义，否则 App 和 Xposed 会对不上。

## 2. 这项目最容易踩的坑

### 2.1 XML / 字符串编码污染
这个仓库历史上有过明显的编码污染，表现为：
- 中文变成 `鍗槦淇℃伅` 这类乱码
- XML 属性被截断，例如：
  - `android:text="80.0绫?`
  - `android:text="12棰?`
- 某些文件即使“肉眼看上去正常”，资源合并时仍然炸掉

**告诫：**
- 改 XML/字符串资源后，一定要重新编译验证
- 如果资源编译报 `Cannot read field "elmName" because "root" is null`，优先怀疑：
  - 文件编码
  - BOM
  - 断裂的 XML 属性
- 尽量统一保存成 **UTF-8 无 BOM**
- 不要在同一个文件里反复用不同工具保存

### 2.2 MIUI 对主题/资源特别挑剔
这个项目在 MIUI 上踩过一个很坑的点：
- `NavigationView` 在某些主题属性配置下会因为颜色资源不是“真正的资源 ID”而崩溃
- 表现为：
  - `Resources$NotFoundException: Resource ID #0x0`
  - 栈在 `NavigationView.createDefaultColorStateList()`

**告诫：**
- Material 组件对 theme attr 的容错不等于 MIUI 有容错
- 有些颜色不要只给“直接 color 值”，要给“稳定资源引用”
- 如果改主题后“打开就闪退”，先查侧边栏、导航、菜单、ColorStateList

### 2.3 不要轻信“看起来能简化”的状态逻辑
比如：
- 路线模拟
- 摇杆模拟
- 单点模拟
- 静止定位稳定模式
- 步频模拟

这些功能之间经常共享一部分状态。

**告诫：**
- 不要因为“变量看起来重复”就合并
- 不要因为“两个按钮都叫开始/停止”就抽象成一个统一状态
- 先确认：
  - 谁控制位置推进
  - 谁控制方向/bearing
  - 谁控制速度
  - 谁在向 Xposed 发命令

否则最容易出现：
- 点了“停止”停不下来
- 路线只到起点但不继续移动
- 摇杆勾选后提示“请先启动模拟”
- 单点静止时仍然抽搐漂移

### 2.4 `SharedPreferences` 历史键名有脏数据
这个仓库曾经长期直接散着用 `SharedPreferences`。

现在已经开始收口到：
- `Portal/app/src/main/java/moe/fuqiuluo/portal/ext/PortalPrefs.kt`
- `Portal/app/src/main/java/moe/fuqiuluo/portal/ext/Perfs.kt`

**告诫：**
- 新配置项优先加到 `PortalConfig + PortalPrefs`
- 旧键名不要轻易改名
- 旧拼写错误的键名也不要直接修正，例如某些历史 key 本身就有拼写错误，贸然改会导致老用户配置丢失

### 2.5 “能编译”不等于“能用”
这个项目有三层验证：
- 编译通过
- App 能打开
- Hook 真正在系统侧生效

**告诫：**
- 改完 UI 只做 Gradle compile，不够
- 至少要做一轮：
  - build
  - install
  - launch
  - logcat
- 如果改的是模拟逻辑，还要验证：
  - 单点模拟是否真改定位
  - 路线模拟是否真移动
  - 停止按钮是否真停止
  - GNSS/基站开关是否生效

## 3. 推荐的修改原则

### 3.1 先保功能，再谈优雅
对这个项目来说：
- “丑但稳定” > “优雅但不兼容”
- “多写几行但不破坏状态” > “抽象很好看但引入时序 bug”

### 3.2 先前台，后底层
如果不是明确要修 Hook：
- 先改 App UI / 字符串 / 设置页 / 配置模型
- 最后再碰 Xposed Hook

### 3.3 新逻辑尽量做成开关
这个项目用户环境复杂、ROM 复杂。

**凡是会改变原行为的改动，尽量：**
- 做成设置开关
- 保留旧行为兜底
- 可以快速回退

例如：
- 静止定位稳定模式
- 路线速度浮动
- 步频模拟
- 禁止 Wi‑Fi 扫描
- 反定位拉回

### 3.4 不要同时大改 UI、配置、协议
最危险的操作是一次改三层：
- 改页面字段
- 改本地配置键名
- 改 `sendExtraCommand` 协议字段

这种改动出了问题会很难定位。

**建议一次只动一层。**

## 4. 推荐的工作流

### 4.1 改资源/UI 时
1. 改布局或字符串
2. 编译：
   - `Portal\gradlew.bat :app:compileArm64DebugKotlin`
3. 如果资源报错：
   - 先查 XML 断裂
   - 再查 BOM / 编码

### 4.2 改 App 行为时
1. 编译：
   - `Portal\gradlew.bat :app:assembleArm64Debug`
2. 安装：
   - `adb install -r C:\Users\31691\Desktop\sandbox\fackeloc\Portal\app\build\outputs\apk\arm64\debug\Portal-v1.0.4.r127.e645722-arm64.apk`
3. 启动：
   - `adb shell am start -n moe.fuqiuluo.portal/.MainActivity`
4. 抓日志：
   - `adb logcat -d AndroidRuntime:E ActivityTaskManager:I WindowManager:I *:S`

### 4.3 改模拟/Hook 行为时
至少验证：
- 首页能打开
- 模拟位置能开始/停止
- 模拟路线能开始/停止
- 摇杆能控制移动
- 设置页开关能同步到系统侧
- logcat 没有新的崩溃和明显报错

## 6. 最不建议做的事

### 不建议 1：直接“全量重写 Xposed 层”
因为：
- 风险极高
- 兼容性经验会全部丢掉
- 很多问题不是代码风格问题，而是 ROM 行为差异

### 不建议 2：为了“整洁”删除旧字段/旧开关
因为：
- 很多旧字段背后可能是兼容旧配置或旧行为
- 你今天觉得“没用”，可能只是还没踩到它的 ROM 场景

### 不建议 3：把多个状态机强行合并
特别是：
- 摇杆状态
- 路线状态
- 模拟启动状态
- 静止/移动判定

### 不建议 4：用一次性脚本批量替换所有中文/颜色/资源
因为：
- 这项目历史文件质量参差不齐
- 很容易把 XML 改成“看着没事，编译直接炸”

---

## 7. 真要重构，正确顺序是什么

推荐顺序：

1. **字符串和资源先收敛**
2. **配置访问集中化**
3. **设置页和 UI 层逐页清理**
4. **状态机拆分：单点 / 路线 / 摇杆 / 步频**
5. **最后再考虑 Compose 化**
6. **最晚才碰 Xposed / Hook 底层**

也就是说：

**先把前台 App 救到可维护，再动系统底层。**

---

## 8. 最后一句忠告

这项目最容易让人犯的错，是把它当成“普通 Android 项目”。

它不是。

它是：
- UI 工具层
- 系统通信层
- Xposed Hook 层
- ROM 兼容层
- 模拟定位行为层

五层叠在一起的项目。

所以正确姿势不是“重写得漂亮”，而是：

**每次只改一小块，始终保持可编译、可安装、可启动、可验证。**

如果你能做到这四点，这坨代码就还有救。
